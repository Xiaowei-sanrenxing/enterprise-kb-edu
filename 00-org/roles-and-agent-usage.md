# 角色与 agent 使用方式

> 本知识库是给 agent 的执行底座，不是要求每个人手动翻文件、手动跑 SOP。
> 人的职责是提出任务、补充事实、审核结果、维护资产；具体执行尽量交给 agent。

## 总原则

1. agent 读库执行：`AGENTS.md`、任务卡、context、SOP、evals 主要服务 agent。
2. 人不手动跑全流程：普通员工不需要自己判断所有上下文、逐项执行工作流。
3. 人负责业务判断：目标、真实素材、授权、价格、客户敏感信息、定版决策仍由人负责。
4. DRI 负责维护：每条工作流的 DRI 负责补充、审核、定版自己工作流的关键资产。

## 老板/负责人怎么用

老板/负责人主要看全局，不做每条 SOP 的手动执行者。

典型动作：

- 让 agent 检查哪条工作流缺资产。
- 让 agent 汇总各工作流风险和待补项。
- 决定 DRI、产品边界、价格权限、对外承诺边界。
- 审核关键资产是否能定版。

示例提问：

```text
读 AGENTS.md 和 INDEX.md，帮我检查现在这套知识库还有哪些关键缺口，按优先级列出来。
```

## DRI/部门主管怎么用

DRI/部门主管负责自己工作流的业务正确性，不需要从零找文件。

典型动作：

- 给 agent 提供真实业务素材。
- 让 agent 按本工作流任务卡生成话术、SOP、复盘或课程材料。
- 审核 agent 输出是否符合业务和权限。
- 决定草稿是否升定版。

示例提问：

```text
读 sales-conversion 的任务卡和 context，基于这段脱敏对话，帮我判断客户分级、下一步话术和流失风险。
```

## 普通员工怎么用

普通员工不应该手动跑完整工作流，而是把具体任务交给 agent，并补充自己知道的业务事实。

普通员工要做：

- 说清楚要完成的具体任务。
- 提供必要素材、客户背景或当前进展。
- 按权限要求隐藏或脱敏敏感信息。
- 检查 agent 输出是否符合实际情况。
- 不确定时请 DRI 审核。

普通员工不要做：

- 自己在全库翻文件拼上下文。
- 自己判断草稿能否对外。
- 把客户隐私发进对外产物。
- 擅自承诺价格、优惠、效果、案例授权。

示例提问：

```text
读 AGENTS.md，判断这件事该走哪条工作流。我要处理一个新线索，请根据我提供的行业、需求和对话记录，生成下一步跟进话术。
```

## 知识库维护者怎么用

知识库维护者负责让库持续变好，而不是亲自替每条业务线做所有判断。

典型动作：

- 检查占位符是否补齐。
- 检查草稿、定版、废弃状态是否清楚。
- 检查权限和隐私风险。
- 更新索引和同步规则。
- 推动 DRI 审核关键资产。

示例提问：

```text
读 AGENTS.md、INDEX.md 和 90-governance，帮我检查当前知识库有哪些占位符、草稿资产、权限风险和待同步内容。
```

## agent 怎么用

agent 必须按 `AGENTS.md` 路由，不要让用户自己承担上下文拼装。

标准动作：

1. 读 `AGENTS.md` 判断工作流。
2. 读对应 `_task-card.md`。
3. 读任务卡列出的 context。
4. 检查权限和定版状态。
5. 执行任务。
6. 对照 evals 自检。
7. 发现新经验时，按治理规则回流。

## 关键边界

- 人给目标、事实、授权和审核。
- agent 读上下文、执行、生成、复盘。
- DRI 维护工作流资产。
- 知识库负责人维护治理、权限、索引和同步。
